Building the ILES Backend with Django REST Framework

The complete, no-shortcuts guide for new programmers. Every concept is fully explained with real code examples, actual JSON output, and step-by-step reasoning. Read this before writing a single line.

Section 1 · Foundation

What is Django REST Framework?

You already know how to build a Django web application — you write views that return HTML pages using templates. Django REST Framework (DRF) changes what your Django app returns. Instead of returning a rendered HTML page, your Django backend now returns raw JSON data. Your React frontend then receives that JSON and decides how to display it.

Here is the mental model you need to hold onto throughout this entire project:

The Restaurant Kitchen Analogy Think of your Django app as a restaurant kitchen. Without DRF, the kitchen cooks the food and plates it beautifully for the customer (Django renders HTML). With DRF, the kitchen only cooks and hands the food in a container (JSON) to a waiter. The waiter (React) then decides how to present it on the table (the browser). The kitchen and the waiter are now two separate, independent workers.

This separation is why React runs on port 3000 and Django runs on port 8000. They are completely separate programs that communicate through a defined contract called an API (Application Programming Interface).

What Does JSON Look Like?

Before anything else, you need to understand JSON — it is the only language your frontend and backend use to talk to each other. JSON is just text formatted as key-value pairs, very similar to a Python dictionary.

Example: What a logbook entry looks like as JSON
{ "id": 5, "week_number": 3, "start_date": "2025-07-14", "end_date": "2025-07-18", "activities": "Worked on database schema design with the team lead.", "status": "pending", "student": 12, "student_username": "john.doe", "submitted_at": "2025-07-19T08:30:00Z" }

When React makes a request to your backend, this is what it gets back. React then reads data.student_username, data.status, etc. and builds the UI from those values. Your job with the backend is to produce clean, correct JSON like this.

The four main pieces you will build are: Models (define database tables), Serializers (convert between Python and JSON), ViewSets (handle the logic), and URLs (map paths to ViewSets). That is it. Everything in DRF is one of these four things.

Section 3 · HTTP Fundamentals

The HTTP Request/Response Cycle

Every single thing that happens in your application starts with an HTTP request from React and ends with an HTTP response from Django. You need to understand this deeply because every error you encounter will be traced back to this cycle.

HTTP methods tell the server what kind of thing the client wants to do. You can think of them as verbs in a sentence where the URL is the noun.

Section 4 · Data Layer

Models: Your Database Blueprint

A Django model is a Python class that defines one database table. Each attribute on the class becomes a column in the table. You already know this from your Django coursework — but let's be very explicit about the models used in ILES so every section that follows makes sense.

Models are your foundation Every serializer, view, and permission you write will reference a model. The model decides what data exists. The serializer decides what of that data to expose. The view decides who can access it.
backend/logbook/models.py — The Logbook model (fully annotated)
from django.db import models
from django.conf import settings  # to reference our custom User model safely


class Logbook(models.Model):
    """
    One row in this table = one weekly logbook entry by a student.
    """

    # STATUS_CHOICES defines the allowed values for the 'status' field.
    # Using a choices tuple like this prevents typos and invalid states.
    # Format: (value_stored_in_DB, human_readable_label)
    STATUS_CHOICES = [
        ('draft',     'Draft'),       # Student is still writing
        ('pending',   'Pending'),     # Student submitted, awaiting review
        ('approved',  'Approved'),    # Supervisor approved it
        ('rejected',  'Rejected'),    # Supervisor sent it back
    ]

    # ForeignKey means "many logbooks can belong to one student".
    # on_delete=CASCADE means if the student is deleted, their logs are deleted too.
    # settings.AUTH_USER_MODEL is safer than importing User directly.
    student = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name='logbooks'   # lets you do: user.logbooks.all()
    )

    placement = models.ForeignKey(
        'placements.Placement',
        on_delete=models.CASCADE,
        related_name='logbooks'
    )

    week_number = models.PositiveIntegerField()   # stores 1, 2, 3... not negative

    start_date = models.DateField()    # YYYY-MM-DD (no time)
    end_date   = models.DateField()

    activities = models.TextField()   # unlimited text (like a TextArea in HTML)

    status = models.CharField(
        max_length=10,
        choices=STATUS_CHOICES,
        default='draft'             # new entries always start as drafts
    )

    submitted_at = models.DateTimeField(null=True, blank=True)  # null until submitted

    # auto_now_add sets to current time ONCE when the record is first created.
    # auto_now updates to current time EVERY TIME the record is saved.
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-created_at']   # newest entries first by default

    def __str__(self):
        return f"Week {self.week_number} — {self.student.username} ({self.status})"

Section 5 · Core Concept

Serializers — The Most Important Concept

A Serializer is the translator between two worlds: the Python/Django world (model instances, querysets) and the JSON world (what React understands). It does two jobs that look like opposites but use the same class:

  • Serialization (outbound — Python → JSON): You have a Logbook Python object from the database. The serializer converts it into a JSON dictionary that can be sent over HTTP.
  • Deserialization (inbound — JSON → Python): React sends you JSON from a form submission. The serializer takes that raw JSON, validates every field, and creates a clean Python object you can safely save to the database.
The Customs Officer Analogy A serializer is like a customs officer at an international airport. When goods (data) leave the country (Django → React), the officer packages everything into standardised JSON "shipping containers." When goods arrive from abroad (React → Django), the officer opens every container, checks that nothing illegal or malformed got through, and only then stamps it through to the database. If anything fails inspection, the officer sends back a 400 error with a precise description of what was wrong.
Section 5b · Deep Dive

Serializer vs ModelSerializer — With Full Examples

DRF gives you two main serializer base classes. Choosing the right one saves or costs you hours of work. Let us look at both for the exact same Logbook data, so you can see the difference clearly.

Option 1: The Base Serializer Class

With serializers.Serializer, you define every single field manually. DRF does not look at your model at all — it only knows what you explicitly tell it. This gives you total control, but it means a lot of repetition.

backend/logbook/serializers.py — using the base Serializer (the hard way)
from rest_framework import serializers

# Notice: we do NOT import the Logbook model here.
# The base Serializer knows nothing about your models.

class LogbookSerializerManual(serializers.Serializer):
    """
    Every field must be declared one-by-one.
    The field type must match your model's field type exactly.
    """
    id           = serializers.IntegerField(read_only=True)
    week_number  = serializers.IntegerField()
    start_date   = serializers.DateField()
    end_date     = serializers.DateField()
    activities   = serializers.CharField()
    status       = serializers.CharField(max_length=10)
    submitted_at = serializers.DateTimeField(read_only=True, allow_null=True)
    created_at   = serializers.DateTimeField(read_only=True)
    updated_at   = serializers.DateTimeField(read_only=True)

    # Because this is the base Serializer, saving is NOT automatic.
    # You must manually write create() and update() methods.

    def create(self, validated_data):
        """
        Called when .save() is called on a new instance.
        You must manually call Logbook.objects.create() yourself.
        """
        from .models import Logbook
        return Logbook.objects.create(**validated_data)

    def update(self, instance, validated_data):
        """
        Called when .save() is called on an existing instance.
        You must manually set each field yourself.
        """
        instance.week_number = validated_data.get('week_number', instance.week_number)
        instance.activities  = validated_data.get('activities',  instance.activities)
        instance.status      = validated_data.get('status',      instance.status)
        instance.save()
        return instance
The problem with the base Serializer Your Logbook model has 10 fields. You wrote 10 lines above, plus a create() method, plus an update() method. That is ~25 lines. Now imagine your model gets a new field — you must remember to add it to the serializer too, or it will silently disappear from your API. This is error-prone.

Option 2: ModelSerializer — What You Will Use 90% of the Time

With serializers.ModelSerializer, you just point it at your model and tell it which fields to include. DRF reads your model definition, figures out the right field types automatically, and generates the create() and update() methods for you.

backend/logbook/serializers.py — using ModelSerializer (the right way)
from rest_framework import serializers
from .models import Logbook

class LogbookSerializer(serializers.ModelSerializer):
    """
    ModelSerializer reads the Logbook model and auto-generates:
      - The correct field type for each model field
      - A working create() method
      - A working update() method

    You only need the Meta class to tell it which model and which fields.
    """

    # This is an EXTRA field that does NOT exist directly on the Logbook model.
    # It "follows" the ForeignKey to the student and gets their username.
    # source='student.username' means: go to the related student object, get .username
    # read_only=True means: include this in responses, but ignore it in incoming POST data
    student_username = serializers.CharField(
        source='student.username',
        read_only=True
    )

    class Meta:
        model  = Logbook   # ← Point to the model

        # List every field you want to expose through the API.
        # Be deliberate — do not expose sensitive fields carelessly.
        fields = [
            'id',
            'week_number',
            'start_date',
            'end_date',
            'activities',
            'status',
            'student',          # the FK integer ID — React can send this when creating
            'student_username', # the extra human-readable field we defined above
            'placement',
            'submitted_at',
            'created_at',
            'updated_at',
        ]

        # These fields will appear in responses but will be IGNORED if
        # React tries to send them in a POST or PATCH request.
        # Django sets them automatically — React has no business touching them.
        read_only_fields = ['id', 'created_at', 'updated_at', 'submitted_at', 'status']

Section 5c · What You Actually See

Serializer Input and Output — Concrete Examples

This is the most important section for understanding what a serializer actually does. Let's trace the exact data transformation in both directions.

Direction 1: Outbound (Python → JSON) — Serialization

Imagine Django has just fetched a Logbook object from the database. It is a Python object. The serializer converts it to a JSON-ready Python dictionary, which DRF then encodes as a string and sends to React.

In your Django shell — python manage.py shell

from logbook.serializers import LogbookSerializer

# Fetch one logbook entry from the database (primary key = 5)
logbook_instance = Logbook.objects.get(pk=5)

# Print the raw Python object — this is what Django has
print(logbook_instance)
# Output: Week 3 — john.doe (draft)

# Now pass it to the serializer. No 'many=True' → serializes a SINGLE object.
serializer = LogbookSerializer(logbook_instance)

# .data is the Python dictionary representation (ready to become JSON)
print(serializer.data)

Direction 2: Inbound (JSON → Python) — Deserialization

React submits a form. Django receives raw JSON. The serializer validates it and — if valid — lets you save it to the database.

In your Django shell — simulating an incoming POST request 
from logbook.serializers import LogbookSerializer

# This is the raw JSON data that came from React's form submission.
# It arrives as a Python dictionary in Django.
incoming_data = {
    "week_number": 3,
    "start_date": "2025-07-14",
    "end_date": "2025-07-18",
    "activities": "Worked on database design with the team lead.",
    "placement": 2,
    # Notice: we do NOT send "student" — the view sets that automatically
    # Notice: we do NOT send "status" — it's read_only, starts as 'draft'
    # Notice: we do NOT send "created_at" — that's read_only too
}

# Pass the data= kwarg (not just the object!) to trigger deserialization mode
serializer = LogbookSerializer(data=incoming_data)

# ALWAYS check is_valid() before doing anything else.
# This triggers all the validation rules.
if serializer.is_valid():
    # All fields passed validation — save to database
    # In views, you'd call serializer.save(student=request.user)
    # In shell, pass the required FK manually:
    logbook = serializer.save(student_id=12)
    print("Saved! New logbook id:", logbook.id)
else:
    # Print the errors — these are what get returned as a 400 response
    print(serializer.errors)

Section 5d · Validation

Validation: Rejecting Bad Data Before It Hits the Database

DRF serializers have three levels of validation, running in this order:

1

Field-Level Type Validation (Automatic)

DRF checks that each field has the right data type. An IntegerField must get a number. A DateField must get a YYYY-MM-DD string. You get this for free.

2

Field-Level Custom Validation (validate_<fieldname>)

Add a method named validate_week_number (or any field name) to add your own business rules for that specific field.

3

Cross-Field Validation (validate())

The validate() method runs after all individual fields pass. Use it when a rule involves comparing two or more fields — like ensuring end_date is after start_date.

backend/logbook/serializers.py — full serializer with all validation types
from rest_framework import serializers
from .models import Logbook

class LogbookSerializer(serializers.ModelSerializer):
    student_username = serializers.CharField(
        source='student.username',
        read_only=True
    )

    class Meta:
        model  = Logbook
        fields = [
            'id', 'week_number', 'start_date', 'end_date',
            'activities', 'status', 'student', 'student_username',
            'placement', 'submitted_at', 'created_at', 'updated_at',
        ]
        read_only_fields = ['id', 'created_at', 'updated_at', 'submitted_at', 'status']

    # ── LEVEL 2: Per-field validation ────────────────────────────────────────
    # DRF automatically calls any method named  validate_
    # The method receives the already type-checked value and must return it
    # (possibly modified) or raise serializers.ValidationError.

    def validate_week_number(self, value):
        """
        Runs after DRF confirms 'value' is an integer.
        Now we add our own business rule: weeks must be 1–52.
        'value' here is the integer that was submitted.
        """
        if value < 1 or value > 52:
            raise serializers.ValidationError(
                "Week number must be between 1 and 52."
            )
        return value  # always return the value if validation passes

    def validate_activities(self, value):
        """
        Activities text must be at least 20 characters.
        'Too short' entries are not useful log entries.
        """
        if len(value.strip()) < 20:
            raise serializers.ValidationError(
                "Please describe your activities in at least 20 characters."
            )
        return value

    # ── LEVEL 3: Cross-field validation ──────────────────────────────────────
    # 'data' is a dictionary containing ALL fields that passed level-1 and level-2.
    # This is your chance to check rules that involve multiple fields together.

    def validate(self, data):
        """
        Cross-field validation.
        Both 'start_date' and 'end_date' are already confirmed as valid DateFields.
        Now we check that they make logical sense relative to each other.
        """
        start = data.get('start_date')
        end   = data.get('end_date')

        if start and end:
            if start >= end:
                # The error key matches the field you want highlighted in React
                raise serializers.ValidationError({
                    "end_date": "End date must be after start date."
                })

            # Also check: the date range should not be more than 7 days (one week)
            from datetime import timedelta
            if (end - start).days > 7:
                raise serializers.ValidationError({
                    "end_date": "A logbook entry can only span one week (7 days max)."
                })

        return data  # always return data if all cross-field checks pass
Output — what React gets back when week_number is 99
# HTTP 400 Bad Request { "week_number": ["Week number must be between 1 and 52."] }
Output — what React gets back when end_date is before start_date
# HTTP 400 Bad Request { "end_date": ["End date must be after start date."] }
Section 6 · Core Concept

Views and ViewSets — In Full Detail

A View is a Python class (or function) that receives an HTTP request, does something with it, and returns an HTTP response. In DRF, there are several types of views, each giving you more or less "for free." Let's understand the full spectrum.

The View Hierarchy — From Manual to Automatic

APIView — The Lowest Level (used for Login)

With APIView you define methods named after HTTP verbs: get(), post(), patch(), delete(). DRF routes requests to the correct method. This is used for the login and register endpoints where there is no direct model CRUD.

backend/users/views.py — RegisterView using APIView
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status, permissions
from .serializers import RegisterSerializer

class RegisterView(APIView):
    """
    Handles: POST /api/auth/register/
    Allows unauthenticated users (anyone can register).
    """
    # Override the default "must be logged in" requirement for this view only
    permission_classes = [permissions.AllowAny]

    def post(self, request):
        """
        DRF calls this method when a POST request hits /api/auth/register/
        request.data is the parsed JSON body — already a Python dict.
        """
        # Pass the incoming data to the serializer to validate it
        serializer = RegisterSerializer(data=request.data)

        if serializer.is_valid():
            # .save() calls the serializer's create() method
            user = serializer.save()

            return Response(
                {"message": f"Account created for {user.username}. You can now log in."},
                status=status.HTTP_201_CREATED   # 201 = Created
            )

        # Validation failed — return the errors dict with a 400 status
        return Response(
            serializer.errors,
            status=status.HTTP_400_BAD_REQUEST
        )
Output — successful registration (201 Created)
{ "message": "Account created for john.doe. You can now log in." }
Output — registration with duplicate username (400 Bad Request)

ModelViewSet — Your Main Tool for Every Resource

A ModelViewSet automatically provides all 6 endpoint actions for a resource. You register it with a Router and the URL patterns are created for you. Let's trace exactly what URLs get created and what code handles each one.

backend/logbook/views.py — Full LogbookViewSet with explanations
from rest_framework import viewsets, permissions, status
from rest_framework.decorators import action
from rest_framework.response import Response
from django.utils import timezone

from .models import Logbook
from .serializers import LogbookSerializer
from users.permissions import IsStudent, IsOwnerOrSupervisor

class LogbookViewSet(viewsets.ModelViewSet):
    """
    When registered with a Router as 'logbook', DRF auto-creates these URLs:

    GET    /api/logbook/          → list()             — get all logbooks
    POST   /api/logbook/          → create()           — create a new logbook
    GET    /api/logbook/{id}/     → retrieve()         — get one specific logbook
    PUT    /api/logbook/{id}/     → update()           — replace a logbook entirely
    PATCH  /api/logbook/{id}/     → partial_update()   — update some fields
    DELETE /api/logbook/{id}/     → destroy()          — delete a logbook

    You do NOT write these methods — ModelViewSet already has them.
    You only override what needs custom behaviour.
    """
    serializer_class   = LogbookSerializer
    permission_classes = [permissions.IsAuthenticated, IsOwnerOrSupervisor]

    def get_queryset(self):
        """
        WHAT: Controls which database rows this user is allowed to see.
        WHY:  Without this, every user could see every logbook entry from everyone.
        HOW:  We filter based on the logged-in user's role.

        self.request.user → the currently authenticated user object
        """
        user = self.request.user

        if user.role == 'student':
            # A student sees ONLY their own logbooks
            return Logbook.objects.filter(student=user)

        elif user.role in ['workplace_supervisor', 'academic_supervisor']:
            # Supervisors see logbooks from students in their placements
            return Logbook.objects.filter(placement__supervisor=user)

        elif user.role == 'internship_admin':
            # Admin sees every logbook in the system
            return Logbook.objects.all()

        # Unknown role — see nothing (safe default)
        return Logbook.objects.none()

    def perform_create(self, serializer):
        """
        WHAT: Called right before a new object is saved to the database.
        WHY:  Students should not be able to create logbooks for other students.
              We force the student field to be the logged-in user.
        HOW:  serializer.save(**kwargs) passes extra fields that weren't in the request.

        Without this: a malicious user could send {"student": 999} in the request body
        and create a logbook pretending to be student 999.
        With this: the student field is always forcibly set to the real requesting user.
        """
        serializer.save(student=self.request.user)

Section 6b · Deep Dive

get_queryset() — Controlling What Data Each User Can See

get_queryset() is one of the most important methods in your entire backend. It answers the question: "Given the user who is making this request, which database rows should they be allowed to retrieve?"

A queryset in Django is like a query you're building up. Logbook.objects.filter(student=user) generates the SQL SELECT * FROM logbook WHERE student_id = 12. It is lazy — Django only runs the actual SQL query when it absolutely has to (when you iterate over it or call .count()).

Why you must override get_queryset() If you do not override it, ModelViewSet uses self.queryset which is typically Logbook.objects.all(). That means every user sees every logbook. Student A can read Student B's logbook. This is a serious data breach. Always override get_queryset().
Example: Queryset for the Evaluation app (Member 4)
from rest_framework import viewsets, permissions
from .models import Evaluation
from .serializers import EvaluationSerializer

            return Evaluation.objects.all()

        # Supervisors can only see evaluations THEY submitted
        if user.role in ['workplace_supervisor', 'academic_supervisor']:

        # Students can see evaluations written about them
        if user.role == 'student':
            return Evaluation.objects.filter(student=user)

        return Evaluation.objects.none()

    def get_permissions(self):
        """
        WHAT: Allows DIFFERENT permission rules for DIFFERENT actions.
        WHY:  A student should be able to READ evaluations (their own)
              but should NOT be able to CREATE evaluations (that is the supervisor's job).

        self.action is a string: 'list', 'create', 'retrieve', 'update',
                                 'partial_update', 'destroy', or a custom action name.

        if self.action == 'create':
            # Only supervisors can create evaluations
            from users.permissions import IsSupervisor
            return [permissions.IsAuthenticated(), IsSupervisor()]

        if self.action == 'destroy':
            # Only admins can delete evaluations
            from users.permissions import IsAdmin
            return [permissions.IsAuthenticated(), IsAdmin()]

        # For list, retrieve — anyone authenticated can try
        return [permissions.IsAuthenticated()]

perform_create() — Auto-Setting Fields on Save

Common use cases in ILES:

  • Force student = request.user — so a student cannot submit a log as someone else
  • Force submitted_by = request.user on evaluations
  • Automatically set the current timestamp for something

from rest_framework import viewsets, permissions
from .models import LogReview
from .serializers import LogReviewSerializer

class LogReviewViewSet(viewsets.ModelViewSet):
    serializer_class   = LogReviewSerializer
    permission_classes = [permissions.IsAuthenticated]

    def get_queryset(self):
        user = self.request.user
        if user.role == 'internship_admin':
            return LogReview.objects.all()
        # Supervisors see reviews they wrote
        if user.role in ['workplace_supervisor', 'academic_supervisor']:
            return LogReview.objects.filter(reviewer=user)
        # Students see reviews on their logbooks
        if user.role == 'student':
            return LogReview.objects.filter(logbook__student=user)
        return LogReview.objects.none()

    def perform_create(self, serializer):
        """
        When a supervisor POSTs to create a review:
        - 'reviewer' is auto-set to whoever is making the request
        - They cannot fake being a different reviewer

        serializer.save(**kwargs) passes keyword arguments as if
        they were sent in the request body — but they're set server-side.
        """
        serializer.save(reviewer=self.request.user)
serializer.save() vs serializer.save(student=request.user) Calling serializer.save() with no arguments just saves what was in the request. Calling serializer.save(student=request.user) overrides or adds the student field with the value from the server. The client's submitted value for student is completely ignored — the server always wins on security-sensitive fields.
Section 6d · Custom Endpoints

Custom @action Endpoints — Beyond Standard CRUD

Sometimes, a standard CRUD operation is not the right fit. For example, "submitting a logbook" is not the same as "updating a logbook" — it is a specific state transition with its own business rules. That is what the @action decorator is for.

An @action adds a completely new URL endpoint to your ViewSet, beyond the standard six. You control the HTTP method, the URL path, who can access it, and exactly what it does.

backend/logbook/views.py — The submit action (added to LogbookViewSet)
from rest_framework.decorators import action
from rest_framework.response import Response
from rest_framework import status
from django.utils import timezone

class LogbookViewSet(viewsets.ModelViewSet):
    # ... (get_queryset and perform_create from before) ...

    @action(
        detail=True,         # True = URL is /api/logbook/{id}/submit/  (needs a specific logbook ID)
                             # False = URL is /api/logbook/submit/      (no ID, acts on the collection)
        methods=['patch'],   # Only PATCH requests are allowed on this URL
        url_path='submit'    # The URL segment added after the ID: /api/logbook/{id}/submit/
    )
    def submit(self, request, pk=None):
        """
        Custom action: PATCH /api/logbook/{id}/submit/

        Purpose: Moves a logbook entry from 'draft' status → 'pending' status.
        Rules:
          1. Only the owner (the student who created it) can submit their own log.
          2. The log must currently be in 'draft' status to be submitted.
          3. Logs that are already 'pending', 'approved', or 'rejected' cannot be re-submitted.

        pk is the logbook ID from the URL: /api/logbook/5/submit/ → pk='5'
        """
        # self.get_object() fetches the Logbook with the given pk.
        # It also automatically checks object-level permissions.
        logbook = self.get_object()

        # Rule 1: Only the owner can submit
        if logbook.student != request.user:
            return Response(
                {"error": "You can only submit your own logbook entries."},
                status=status.HTTP_403_FORBIDDEN
            )

        # Rule 2: Must be in draft state
        if logbook.status != 'draft':
            return Response(
                {"error": f"Cannot submit a logbook that is already '{logbook.status}'."},
                status=status.HTTP_400_BAD_REQUEST
            )

        # All rules passed — perform the state transition
        logbook.status       = 'pending'
        logbook.submitted_at = timezone.now()
        logbook.save()

        # Return the updated logbook data so React can refresh its display
        serializer = LogbookSerializer(logbook)
        return Response(serializer.data, status=status.HTTP_200_OK)

    @action(detail=True, methods=['patch'], url_path='approve')
    def approve(self, request, pk=None):
        """
        Custom action: PATCH /api/logbook/{id}/approve/
        Only supervisors should be able to call this (permission checked in get_permissions).
        """
        logbook = self.get_object()

        if logbook.status != 'pending':
            return Response(
                {"error": "Only pending logbooks can be approved."},
                status=status.HTTP_400_BAD_REQUEST
            )

        logbook.status = 'approved'
        logbook.save()

        return Response(LogbookSerializer(logbook).data)

    @action(detail=True, methods=['patch'], url_path='reject')
    def reject(self, request, pk=None):
        """
        Custom action: PATCH /api/logbook/{id}/reject/
        """
        logbook = self.get_object()
        feedback = request.data.get('feedback', '')

        if not feedback:
            return Response(
                {"feedback": "Please provide feedback explaining the rejection."},
                status=status.HTTP_400_BAD_REQUEST
            )

        logbook.status = 'rejected'
        logbook.save()

        # Optionally create a review record with the feedback
        # (Member 5's reviews app would handle this)
        return Response(LogbookSerializer(logbook).data)

Section 8 · Security

Permissions — Who Is Allowed to Do What

A Permission class answers a yes/no question: "Is this user allowed to do this specific thing?" DRF checks permissions before running any view logic. If the check fails, the user immediately gets a 403 Forbidden response and the code inside your view never runs.

Two Types of Permissions

has_permission(request, view)

View-level. Checks before any DB query. "Is this user allowed to access this endpoint at all?" Example: only supervisors can reach the evaluate endpoint.

has_object_permission(request, view, obj)

Object-level. Checks after the specific object is fetched. "Is this user allowed to access this specific logbook?" Example: a student can only read their own logbook, not others'.

Important: object-level permissions are only called for detail views has_object_permission is only triggered when a specific object is fetched — meaning endpoints with an ID in the URL, like /api/logbook/5/. For GET /api/logbook/ (the list), only has_permission runs. That is why get_queryset() filtering is also necessary.
backend/users/permissions.py — All ILES permission classes
from rest_framework.permissions import BasePermission, SAFE_METHODS

class IsStudent(BasePermission):
    """
    Grants access only to users whose role is 'student'.

    has_permission runs first, before any object is fetched.
    It decides: can this user access this endpoint TYPE at all?
    """
    def has_permission(self, request, view):
        # request.user.is_authenticated → True if the user has a valid JWT token
        # If they're not authenticated, we never even check the role
        return (
            request.user.is_authenticated and
            request.user.role == 'student'
        )


class IsSupervisor(BasePermission):
    """Grants access to workplace supervisors and academic supervisors."""
    def has_permission(self, request, view):
        return (
            request.user.is_authenticated and
            request.user.role in ['workplace_supervisor', 'academic_supervisor']
        )


class IsAdmin(BasePermission):
    """Grants access only to internship administrators."""
    def has_permission(self, request, view):
        return (
            request.user.is_authenticated and
            request.user.role == 'internship_admin'
        )


class IsOwnerOrReadOnlySupervisor(BasePermission):
    """
    Object-level permission.

    Rules:
    - Admins can do anything.
    - Students can only modify their own logbooks.
    - Supervisors can READ logbooks from their placements, but not modify them.
      (Supervisors modify via the review/approval @actions, not raw CRUD)

    SAFE_METHODS = ('GET', 'HEAD', 'OPTIONS') — read-only requests
    """
    def has_object_permission(self, request, view, obj):
        user = request.user

        # Admin overrides everything
        if user.role == 'internship_admin':
            return True

        # Students own their logbooks
        if user.role == 'student':
            return obj.student == user  # True only if this logbook belongs to them

        # Supervisors can READ (GET) logbooks in their supervised placements
        if user.role in ['workplace_supervisor', 'academic_supervisor']:
            if request.method in SAFE_METHODS:
                return obj.placement.supervisor == user
            return False  # Supervisors cannot PUT/PATCH/DELETE raw logbook data

        return False   # Unknown role → deny

Using Multiple Permissions — AND vs OR Logic

When you set permission_classes = [PermissionA, PermissionB], DRF checks both. The user must pass ALL of them. This is AND logic.

Section 9 · Authentication

JWT Authentication — In Full Detail

Traditional Django uses sessions: when you log in, Django saves a session record in the database and gives you a cookie. Your browser sends that cookie automatically with every request. This works great for server-rendered HTML, but it does not work well for APIs.

For APIs, we use JSON Web Tokens (JWT). Instead of a session, the server gives you a signed token string. You store it yourself and send it manually with every request. The server does not need a database lookup to verify it — it just checks the signature.

Anatomy of a JWT Token

A JWT looks like three Base64-encoded strings joined by dots:

A real JWT token (broken into parts for readability) 
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9    ← Part 1: Header (encoding algorithm)
.eyJ1c2VyX2lkIjoxMiwicm9sZSI6InN0dWRlbnQiLCJleHAiOjE3MjE5MTI2MDB9  ← Part 2: Payload (your data)
.8kLmN3hx9sVz2_VQb0pAz7gJfLTx2x5ARyemdqmqrQ   ← Part 3: Signature (tamper-proof)

Part 2 (the Payload) is just Base64-encoded JSON. Anyone can decode it and read it. The secret is in Part 3 — it is signed with a server-side secret key. If anyone tampers with Part 2, Part 3 becomes invalid and the server rejects it.

Decoded Payload — what is actually inside a JWT for ILES
{ "token_type": "access", "exp": 1721912600, ← Unix timestamp: token expires at this time "iat": 1721909000, ← Unix timestamp: token was issued at this time "jti": "a3b2c1...", ← unique token ID "user_id": 12, ← which user this belongs to "role": "student", ← role (added via custom JWT config) "username": "john.doe" ← username (added via custom JWT config) }
Two tokens — why? The access token is short-lived (60 min). If stolen, the attacker can only use it for one hour. The refresh token is long-lived (7 days) and stored more carefully. When the access token expires, React uses the refresh token to silently get a new access token without making the user log in again. The refresh token is never sent on every request — only to the /api/auth/refresh/ endpoint.

Section 10 · Frontend–Backend Bridge

CORS: Why Your Browser Blocks Requests

Browsers have a built-in security feature called the Same-Origin Policy. It says: a webpage loaded from one origin (combination of protocol + domain + port) cannot make requests to a different origin. This exists to prevent malicious websites from silently reading your bank account data, for example.

Your React app runs at http://localhost:3000. Your Django API runs at http://localhost:8000. These are different origins (different port numbers). The browser treats them as completely separate, like separate countries.

You WILL see this exact error without CORS setup Access to fetch at 'http://localhost:8000/api/logbook/' from origin 'http://localhost:3000' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.

This is NOT a code bug. Your code is correct. It is a missing configuration. Django needs to explicitly say: "I allow requests from port 3000."

The fix is the django-cors-headers package. It adds special HTTP response headers that tell the browser: "This server allows cross-origin requests from localhost:3000." The browser sees those headers and allows the request through.

How CORS Headers Work

When React tries to make a cross-origin request, the browser first sends a preflight OPTIONS request to Django asking "Do you allow this?" Django responds with CORS headers saying yes, and only then does the browser send the real request.

Section 12 · Frontend Connection

Connecting React to Django — The Complete api.js

On the frontend, every API call goes through a single service file: api.js. This is the only place in React that talks to the backend. All components import functions from here — they never call fetch() directly. This makes it easy to change the backend URL in one place.

frontend/src/services/api.js — complete annotated version
// The base URL for ALL backend requests.
// In development, Django runs on port 8000.
// When you deploy to production, change this to your real domain.
const BASE_URL = 'http://localhost:8000/api';

// ── HELPERS ──────────────────────────────────────────────────────────────────

/**
 * getAuthHeaders()
 *
 * Every protected API call needs an Authorization header with the JWT token.
 * This function builds that header by reading the token from localStorage.
 *
 * If no token exists (user is not logged in), the Authorization header
 * is simply omitted — the backend will return 401 Unauthorized.
 */
function getAuthHeaders() {
  const token = localStorage.getItem('access_token');
  return {
    'Content-Type': 'application/json',
    // The spread operator (...) conditionally adds the Authorization header
    // only when a token exists. If token is null, the object spread adds nothing.
    ...(token && { 'Authorization': `Bearer ${token}` }),
  };
}

/**
 * apiRequest(endpoint, options)
 *
 * A wrapper around the browser's built-in fetch() that:
 *   1. Prepends the base URL automatically
 *   2. Adds authentication headers to every request
 *   3. Handles token expiry (401 errors) centrally — redirects to login
 *   4. Throws errors with helpful messages from the server
 *   5. Returns the parsed JSON data on success
 *
 * All other functions in this file call apiRequest() instead of fetch() directly.
 */
async function apiRequest(endpoint, options = {}) {
  const url = `${BASE_URL}${endpoint}`;

  // Merge the auth headers with any options passed in (like method: 'POST', body: ...)
  // Options provided by the caller take priority over the defaults.
  const config = {
    headers: getAuthHeaders(),
    ...options,
    // If the caller also provides headers, merge them with the auth headers
    headers: { ...getAuthHeaders(), ...(options.headers || {}) },
  };

  const response = await fetch(url, config);

  // 401 = our token expired or was never sent
  // Clear the stored tokens and redirect to login page.
  // This handles token expiry automatically for every API call.
  if (response.status === 401) {
    localStorage.removeItem('access_token');
    localStorage.removeItem('refresh_token');
    window.location.href = '/login';
    return;  // stop execution — the page is redirecting
  }

  // Parse the JSON body of the response
  // .catch(() => ({})) handles edge cases where the response has no JSON body
  const data = await response.json().catch(() => ({}));

  // If the HTTP status is not in the 200-299 range, it is an error.
  // Throw an Error so the calling code can catch it in a try/catch block.
  if (!response.ok) {
    // Try to extract a helpful error message from the response body.
    // DRF error responses look like: {"detail": "..."} or {"field": ["error"]}
    const message =
      data.detail ||
      data.error ||
      Object.values(data).flat().join(', ') ||
      `HTTP ${response.status} error`;
    throw new Error(message);
  }

  return data;  // Return the parsed JSON to the calling function
}

// ── AUTH ENDPOINTS ───────────────────────────────────────────────────────────

/**
 * loginUser({ username, password })
 *
 * Sends credentials to Django. On success:
 *   - Stores both tokens in localStorage
 *   - Fetches the user profile (to get role, name, etc.)
 *   - Returns { user, token } to the calling component
 */
export async function loginUser({ username, password }) {
  const data = await apiRequest('/auth/login/', {
    method: 'POST',
    body: JSON.stringify({ username, password }),
  });
  // data = { access: "eyJ...", refresh: "eyJ..." }

  localStorage.setItem('access_token',  data.access);
  localStorage.setItem('refresh_token', data.refresh);

  // Immediately fetch the user's profile so we have their role etc.
  const user = await getProfile();
  return { user, token: data.access };
}

/**
 * getProfile()
 * Returns the currently logged-in user's data.
 * DRF reads the JWT token and knows who you are.
 */
export async function getProfile() {
  return apiRequest('/auth/profile/');
}

/**
 * logoutUser()
 * Clears tokens from localStorage. Next request will get a 401 and redirect.
 * We do NOT need to tell the server — JWT tokens are stateless.
 */
export function logoutUser() {
  localStorage.removeItem('access_token');
  localStorage.removeItem('refresh_token');
}

How a React Component Uses api.js

frontend/src/pages/LogbookPage.jsx — example usage
import { useEffect, useState } from 'react';
import { getLogbooks, submitLogbook } from '../services/api';

function LogbookPage() {
  const [logbooks, setLogbooks] = useState([]);
  const [error, setError]       = useState(null);
  const [loading, setLoading]   = useState(true);

  // Fetch logbooks when the component mounts
  useEffect(() => {
    async function fetchData() {
      try {
        const data = await getLogbooks();   // calls GET /api/logbook/
        setLogbooks(data);
      } catch (err) {
        setError(err.message);  // err.message is the string we threw in apiRequest
      } finally {
        setLoading(false);
      }
    }
    fetchData();
  }, []);

  async function handleSubmit(logbookId) {
    try {
      const updated = await submitLogbook(logbookId);  // PATCH /api/logbook/5/submit/
      // Replace the old logbook in state with the updated version
      setLogbooks(prev => prev.map(lb => lb.id === updated.id ? updated : lb));
    } catch (err) {
      alert(err.message);  // "Cannot submit a logbook that is already 'pending'."
    }
  }

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error: {error}</p>;

  return (
    <ul>
      {logbooks.map(lb => (
        <li key={lb.id}>
          Week {lb.week_number} — Status: {lb.status}
          {lb.status === 'draft' && (
            <button onClick={() => handleSubmit(lb.id)}>Submit</button>
          )}
        </li>
      ))}
    </ul>
  );
}

export default LogbookPage;

Section 15 · Reference

Vocabulary Glossary

Serializer

Converts Python model objects to JSON (outbound) and validates incoming JSON before saving (inbound). The gatekeeper between your database and the outside world.

ModelSerializer

A serializer that reads your model definition and auto-generates fields, create(), and update(). Use this instead of the base Serializer for anything backed by a model.

ViewSet

A class that bundles all CRUD operations for a resource into one place. ModelViewSet provides all 6 operations for free.

Router

Automatically generates URL patterns from a ViewSet registration. One router.register() line replaces 6+ url patterns.

Permission

A rule class that answers yes/no: "Can this user do this thing?" Checked before any view logic runs. Returning False results in a 403.

JWT (JSON Web Token)

A signed string that proves who you are. Has three parts: header, payload (user data + expiry), signature. The server verifies the signature — it cannot be faked.

Access Token

Short-lived JWT (60 min). Sent in the Authorization: Bearer header with every API request. Expires often to limit theft damage.

Refresh Token

Long-lived JWT (7 days). Used ONLY to obtain a new access token when the current one expires. Never sent on regular API calls.

CORS

Browser security policy blocking requests from one origin to another. Solved by adding django-cors-headers to Django and configuring CORS_ALLOWED_ORIGINS.

Queryset

A lazy database query in Django. Logbook.objects.filter(student=user) is a queryset. Django only runs the SQL when you iterate over it or call .count().

get_queryset()

Override this in a ViewSet to control which rows a user can see. Without it, all users see all rows — a data breach.

perform_create()

Override this to force-set fields before saving (e.g., student=request.user). Prevents users from faking ownership of records.

@action

Decorator that adds a custom endpoint to a ViewSet beyond standard CRUD. Used for state transitions like "submit", "approve", "reject".

request.user

The authenticated user object, automatically set by DRF when a valid JWT token is in the request headers. Your views use this constantly.

serializer.data

The serialized Python dictionary, ready to be turned into JSON. Read this after calling serializer.is_valid() or on a freshly constructed serializer.

validated_data

The cleaned, validated dictionary available after serializer.is_valid() returns True. Safe to use for creating/updating database records.
